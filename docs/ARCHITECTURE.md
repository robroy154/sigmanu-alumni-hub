## 1. Project Overview

The Sigma Nu Mu Xi Chapter Alumni Hub is a production web application serving the chapter's alumni community at Columbus State University. The immediate driver is the fraternity's 30th anniversary reunion event, but the site is designed as a **permanent, reusable platform** for alumni contact management and future event hosting.

### 1.1 Core Features

| Feature | Description |
|---|---|
| Event Registration | Alumni register for chapter events, with payment processed via Stripe |
| Authentication & Profiles | Alumni create accounts, manage contact info, verified by admin before full access |
| Big/Little Family Tree | Dynamic visual graph of the full chapter lineage across all generations |
| Alumni Directory | Authenticated members can view the full member directory |
| Admin Panel | 2-3 admins manage registrations, profiles, relationships, approvals, and announcements |

### 1.2 Target Scale

| Metric | Value |
|---|---|
| Expected registered accounts | 150–300 alumni |
| Event registrants (initial) | 150–300 per event |
| Admin users | 2–3 |
| Timeline | As soon as possible |
| Chapter | Sigma Nu, Mu Xi — Columbus State University |

## 2. Finalized Stack

> All stack decisions are locked. No layer should be swapped without documenting the reason and updating this file and `CLAUDE.md`.

| Layer | Decision |
|---|---|
| Framework | Next.js 14+ (App Router) |
| Language | TypeScript |
| Database | Supabase (managed PostgreSQL) |
| Auth | Supabase Auth — email/password + Google, Facebook, Apple OAuth |
| File Storage | Supabase Storage (profile photos) |
| Family Tree Visualization | React Flow |
| Payments | Stripe (2.9% + $0.30/transaction, no monthly fee) |
| Hosting | Vercel |
| Version Control | GitHub |
| Styling | Tailwind CSS + shadcn/ui |

### 2.1 Stack Rationale

**Next.js**
Single codebase for frontend and API routes. Server-side rendering is critical for the family tree initial load — a 200–300 node graph rendered entirely client-side produces slow first paint. App Router provides clean auth-protected routing patterns.

**Supabase**
Fully managed Postgres with zero ops overhead. Handles auth (including all three social providers), file storage for profile photos, and exposes a clean admin dashboard. Free tier comfortably covers expected scale. First paid tier is $25/month if needed.

**React Flow**
Purpose-built for node graphs in React. Handles pan, zoom, click interaction, and minimap natively. Performant at 200–300 nodes. D3.js was considered and rejected — more powerful but significantly more build and maintenance overhead for this use case.

**Vercel**
Native Next.js hosting. Automatic SSL, GitHub-connected deployment on every push, generous free tier. No server to manage.

**Stripe**
Cleaner developer experience than PayPal. Better embedded checkout UX. Cheaper per transaction (2.9% + $0.30 vs PayPal's 3.49% + $0.49). No monthly fees. Card data never touches the application server — Stripe handles PCI compliance.

## 3. Data Model

All tables live in Supabase (PostgreSQL). Column names use `snake_case` throughout. This section is the canonical schema reference.

### 3.1 `members`

The central table. Every person in the system — pending, approved, or admin — is a member record. The `big_id` self-referencing foreign key drives the entire family tree.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key, generated by Supabase Auth |
| `email` | `text` | NO | Unique. Used for login. |
| `first_name` | `text` | NO | Required on signup |
| `last_name` | `text` | NO | Required on signup |
| `nickname` | `text` | YES | Fraternity name |
| `pledge_class` | `text` | YES | Selected from ordered enum (see Section 3.6) |
| `pin_number` | `text` | YES | Admin-editable only |
| `phone` | `text` | YES | |
| `city` | `text` | YES | |
| `state` | `text` | YES | |
| `street_address` | `text` | YES | Full street line (e.g. "123 Main St") |
| `zip` | `text` | YES | Postal code |
| `country` | `text` | NO | Default: `'USA'` |
| `birthday` | `date` | YES | ISO date. Stored as DATE, serialised as `YYYY-MM-DD` string via JS. |
| `home_address` | `text` | YES | Legacy — deprecated in favour of structured address fields |
| `linkedin_url` | `text` | YES | |
| `profile_photo_url` | `text` | YES | Points to Supabase Storage object |
| `show_phone` | `boolean` | NO | Default: `true`. Privacy toggle — hide phone from non-admin members. |
| `show_address` | `boolean` | NO | Default: `true`. Privacy toggle — hide address from non-admin members. |
| `show_birthday` | `boolean` | NO | Default: `true`. Privacy toggle — hide birthday from non-admin members. |
| `referred_by` | `uuid` | YES | FK → members.id. Admin-only: SELECT revoked from authenticated role. |
| `big_id` | `uuid` | YES | FK → members.id. Null for founding/rootless members. |
| `status` | `text` | NO | Enum: `pending` \| `member` \| `admin`. Default: `pending`. |
| `created_at` | `timestamptz` | NO | Auto-set |

### 3.2 `badges`

Admin-assigned recognition only. Members cannot assign their own badges.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `member_id` | `uuid` | NO | FK → members.id |
| `badge_type` | `text` | NO | e.g. `Alumni Chapter Officer`, `Advisory Board` |
| `assigned_by` | `uuid` | NO | FK → members.id (the assigning admin) |
| `assigned_at` | `timestamptz` | NO | Auto-set |

### 3.3 `events`

One record per event. Admin creates and manages. Reusable for future events — no code changes required to spin up a new event. Multiple events can be published simultaneously; the homepage hero shows the next upcoming published event by date.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `title` | `text` | NO | e.g. `30th Anniversary Reunion` |
| `description` | `text` | YES | Rich text or plain text |
| `event_date` | `timestamptz` | NO | |
| `location` | `text` | YES | Venue name and address |
| `ticket_price` | `numeric(10,2)` | NO | `0.00` for free events |
| `capacity` | `integer` | YES | Null = unlimited |
| `status` | `event_status` | NO | Enum: `draft` \| `published` \| `archived`. Default: `draft`. |
| `registration_open` | `boolean` | NO | Controls whether registration CTA buttons render on `/events/[id]`. Set independently of `status` — an event can be `published` (visible) but have `registration_open = false` (no registration buttons). |
| `created_at` | `timestamptz` | NO | Auto-set |
| `updated_at` | `timestamptz` | NO | Auto-updated via trigger |

**Status semantics:**

- `draft` — not visible to the public; work in progress
- `published` — visible at `/events/[id]`, open for registration
- `archived` — hidden from all public routes; registration history preserved, record never deleted

**Canonical routes:**

- `/events/[id]` — public event detail page (unauthenticated accessible); shows registration CTAs only when `registration_open = true`
- `/events/[id]/register` — alumni registration form (auth required); lives in `(alumni)` route group so its auth layout does not block the guest sibling
- `/events/[id]/register/guest` — public guest registration form (no auth); uses admin client for all DB writes
- `/events/[id]/register/guest/confirmation` — public guest confirmation; verifies payment via Stripe session or reads registration_id directly for free events
- `/admin/events` — admin list, create, edit, archive

### 3.4 `registrations`

One record per person per event. `member_id` is nullable to support pending-approval users who have registered but not yet been granted full access, and to allow non-member guests. Guest rows always have `member_id = null`; duplicate prevention for guests is enforced by unique `email + event_id` check in the server action (not a DB constraint).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `event_id` | `uuid` | NO | FK → events.id |
| `member_id` | `uuid` | YES | FK → members.id. Null if unverified registrant. |
| `registrant_name` | `text` | NO | Full name captured at registration |
| `email` | `text` | NO | |
| `phone` | `text` | YES | |
| `dietary_restrictions` | `text` | YES | |
| `tshirt_size` | `text` | YES | Enum: `S` \| `M` \| `L` \| `XL` \| `XXL` |
| `guest_count` | `integer` | NO | Default: `0`. Incremented when guests added post-payment. |
| `payment_status` | `text` | NO | Enum: `unpaid` \| `paid` \| `refunded`. Default: `unpaid`. |
| `stripe_payment_id` | `text` | YES | Stripe payment reference for original registration payment only. New payments go to `registration_payments`. |
| `pending_guests` | `jsonb` | YES | Temporarily holds new guest names during add-guests Stripe checkout. Written by `addGuestsToRegistration`, cleared by webhook after payment. Null at rest. |
| `submitted_at` | `timestamptz` | NO | Auto-set |

### 3.5 `referrals`

One record per invite sent. Token is a unique UUID auto-generated by the DB default. Status progresses `pending` → `completed` (when invitee signs up) or `expired` (nightly pg_cron job). Members can only read their own referrals (RLS). All mutations (completing, expiring) use the service role.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `referred_by` | `uuid` | NO | FK → members.id (the inviting member) |
| `first_name` | `text` | NO | Invitee first name (set by referrer) |
| `last_name` | `text` | NO | Invitee last name (set by referrer) |
| `email` | `text` | NO | Invitee email (set by referrer, read-only on join form) |
| `token` | `uuid` | NO | Unique. Used in `/join?token=` URL |
| `status` | `referral_status` | NO | Enum: `pending` \| `completed` \| `expired`. Default: `pending` |
| `created_at` | `timestamptz` | NO | Auto-set |
| `completed_at` | `timestamptz` | YES | Set when invitee creates their account |
| `expires_at` | `timestamptz` | NO | `created_at + 7 days` |

### 3.6 `announcements`

Admin-posted announcements displayed on the authenticated homepage. Admins write via admin client (service role); members see active rows via RLS (`is_active = true`).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `title` | `text` | NO | Short headline |
| `body` | `text` | NO | Full announcement content |
| `is_active` | `boolean` | NO | Default: `true`. Set false to hide without deleting |
| `created_by` | `uuid` | NO | FK → members.id (admin who posted) |
| `created_at` | `timestamptz` | NO | Auto-set |
| `updated_at` | `timestamptz` | NO | Auto-set; updated on toggle |

### 3.7 `registration_guests`

Individual guest names stored as separate rows rather than a comma-separated blob. Keeps exports clean and data queryable.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `registration_id` | `uuid` | NO | FK → registrations.id |
| `guest_name` | `text` | NO | |

### 3.8 `registration_payments`

Records each post-registration guest payment separately. The original registration payment is tracked only on `registrations.stripe_payment_id`; this table is for add-guest payments only. Inserts via service role (webhook); members can read their own via RLS.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | `uuid` | NO | Primary key |
| `registration_id` | `uuid` | NO | FK → registrations.id (cascade delete) |
| `stripe_payment_id` | `text` | NO | Stripe PaymentIntent id |
| `amount` | `numeric(10,2)` | NO | Total charged. Check: `> 0`. |
| `guest_count_delta` | `integer` | NO | Number of guests added. Check: `> 0`. |
| `created_at` | `timestamptz` | NO | Auto-set |

### 3.6 Pledge Class Enum

`pledge_class` is stored as a `text` field on the `members` table, selected from a programmatically generated ordered list. The sequence is:

- **Single-letter classes first:** Alpha, Beta, Gamma, Delta, Epsilon, Zeta, Eta, Theta, Iota, Kappa, Lambda, Mu, Nu, Xi, Omicron, Pi, Rho, Sigma, Tau, Upsilon, Phi, Chi, Psi, Omega (24 classes)
- **Double-letter classes next:** Beta Alpha, Beta Beta, Beta Gamma ... Beta Omega, then Gamma Alpha ... all the way to Omega Omega (576 classes)
- **Total:** 600 possible pledge classes — sufficient for ~300 years at one class per semester
- **Note:** The double-letter sequence starts at Beta Alpha, not Alpha Alpha. Alpha Alpha does not exist.
- **Implementation:** The enum list is generated programmatically at build time. No one types pledge class names manually.
- **No year field needed:** Pledge class names never repeat within the sequence, so no disambiguation is required.

### 3.7 Family Tree Query Patterns

The `big_id` self-referencing FK enables the following query patterns:

```sql
-- Get someone's Littles
SELECT * FROM members WHERE big_id = '[member_id]';

-- Get someone's Big
-- Resolved via the big_id field on their own record

-- Walk full lineage (up or down) — recursive CTE
WITH RECURSIVE lineage AS (
  SELECT * FROM members WHERE id = '[member_id]'
  UNION ALL
  SELECT m.* FROM members m
  JOIN lineage l ON m.big_id = l.id
)
SELECT * FROM lineage;

-- Get all root nodes (founding/rootless members)
SELECT * FROM members WHERE big_id IS NULL;

-- Full tree for visualization (single query, entire adjacency list)
SELECT id, first_name, last_name, nickname, pledge_class,
       profile_photo_url, big_id
FROM members
WHERE status IN ('member', 'admin')
ORDER BY pledge_class;
```

## 4. Access Control Model

Four distinct access states. `status` is stored on the `members` table. All route-level enforcement is handled in Next.js middleware — never client-side only.

| Status / State | What They Can Access |
|---|---|
| **Unauthenticated (public)** | Event landing page only. No registration, no directory, no tree. |
| **`pending`** | Event registration form and confirmation. Cannot access directory, family tree, or profiles. Cannot see other members. |
| **`member`** | Full access: directory, family tree, own profile editing, event registration, member profiles. |
| **`admin`** | Everything a member can access, plus the full admin panel. |

### 4.1 Admin Capabilities

- View and export all registrations (CSV / Excel)
- Edit or delete individual registrations
- View and export full member contact list
- Edit or delete member profiles
- Manage big/little relationships on behalf of members
- Approve or deny pending account registrations
- Assign and remove badges
- Send email notifications and announcements
- Create, edit, and manage events
- Toggle registration open/closed per event

### 4.2 Profile Visibility Rules

- All profile fields are optional except: `first_name`, `last_name`, `email`, `pledge_class`
- Authenticated members (`member` or `admin`) can view all other profiles
- Individual members choose what optional fields to fill — empty fields simply do not display
- Badges are display-only for regular members, editable only by admin
- `pin_number` is visible to the member themselves and admins only

## 5. Family Tree Visualization

### 5.1 Behavior

| Behavior | Detail |
|---|---|
| Default view | Full chapter tree rendered on load, zoomed to fit the viewport |
| Navigation | Pan and zoom freely. Minimap in corner for orientation. |
| Search | Search bar flies the viewport to the matched member's node |
| Node click | Opens that member's full profile in a side panel or modal |
| Auth required | Login required to view. Not accessible to unauthenticated users. |

### 5.2 Node Display

Each node in the graph shows:
- Full name
- Pledge class
- Profile photo (avatar placeholder if none uploaded)
- Fraternity nickname

### 5.3 Mobile

Desktop is the primary use case. Mobile is nice-to-have and will not block launch. Basic pan/zoom touch support is included by default in React Flow. Deep mobile optimization is a post-launch item.

### 5.4 Lineage Naming Convention

The chapter uses the standard generational naming convention:

| Relationship | Name |
|---|---|
| Direct | Big Brother / Little Brother |
| Two generations | Grandbig / Grandlittle |
| Three generations | Great-Grandbig / Great-Grandlittle |
| And so on... | Mirrors American generational grandparent naming |

This naming is used in display labels and profile lineage sections. It is not stored in the data model.

## 6. Payment Architecture

### 6.1 Provider

Stripe. Integrated directly into the application. No third-party checkout platforms (Eventbrite, Venmo, etc.).

### 6.2 Payment Flow

1. User completes registration form and submits
2. Application creates a `pending` registration record in the database
3. Application creates a Stripe Checkout session with ticket price and registration ID as metadata
4. User is redirected to Stripe Checkout — card data never touches the application server
5. Stripe processes payment and sends a webhook to the application
6. Webhook handler updates registration: `payment_status → paid`, `stripe_payment_id` stored
7. User is redirected to a confirmation page

### 6.3 Stripe Fee Structure (US, as of April 2026)

| Scenario | Fee |
|---|---|
| Standard domestic card | 2.9% + $0.30 per transaction |
| International card | 4.4% + $0.30 (additional 1.5%) |
| Chargeback/dispute | $15 per dispute (non-refundable) |
| Monthly fee | $0 |
| Example: $75 ticket | ~$2.48 in Stripe fees |
| Example: 200 registrants at $75 | ~$496 total in fees |

### 6.4 Implementation Notes

- Use Stripe Checkout (hosted) — not Stripe Elements — for the initial build. Simpler, PCI-compliant by default, can be customized later.
- Always verify payment via webhook, never via client-side redirect alone. Client redirects can be spoofed.
- Store `stripe_payment_id` on the registration record for all refund and dispute handling.
- Refunds are processed through the Stripe dashboard by admin, not through the application UI (at least for v1).

## 7. Recommended Build Order

Build in this sequence. Each phase produces something functional before moving to the next. Do not skip ahead.

| Phase | Deliverable |
|---|---|
| **1 — Project scaffolding** | Next.js + TypeScript + Tailwind initialized, pushed to GitHub, deployed to Vercel (empty shell) |
| **2 — Supabase setup** | Database schema created, environment variables connected, Supabase client initialized, `schema.sql` committed |
| **3 — Auth** | Signup, login, logout, Google/Facebook/Apple OAuth, `pending` status on new accounts, admin approval flow |
| **4 — Member profiles** | Profile creation, editing, photo upload to Supabase Storage, profile view page |
| **5 — Event registration** | Event display page (public), registration form (pending + member), Stripe payment integration, confirmation |
| **6 — Admin panel** | Registration management, member management, account approvals, badge assignment, CSV export |
| **7 — Directory** | Member directory page (authenticated only), search and filter by name/pledge class |
| **8 — Family tree** | Big/little relationship management on profiles, React Flow visualization, minimap, search-to-fly |
| **9 — Email notifications** | Admin announcement sending, registration confirmation emails, big brother update alerts, little brother claim notifications |
| **10 — Polish** | Sigma Nu visual theming, mobile cleanup, performance review, final QA |

**Why this order:**
- Auth first because everything else depends on knowing who the user is
- Registration before the family tree because the event is the immediate priority
- Family tree last because it's the most complex feature and benefits from having real member data already in the database to test with

## 9. Visual Design and Branding

### 9.1 Design Direction

- **Reference:** sigmanu.org for official insignia, colors, and design guidelines
- **Colors:** Sigma Nu Black, White, and Gold are the official palette. Navy is not an official Sigma Nu color.
- **Aesthetic:** Clean and modern — not dated fraternity website energy
- **Graphics:** Placeholders provided in the build for custom assets (crest, chapter photo, event banner). Rob supplies actual graphics.
- **Feel:** Should look like it belongs to this fraternity, not like a generic alumni portal template

### 9.2 Color Tokens (globals.css @theme inline)

Tailwind v4 uses CSS-based token definitions in `globals.css` — there is no `tailwind.config.ts` color map.

```css
/* @theme inline block in src/app/globals.css */
--color-sn-black:           #0b0b0c;   /* Primary background */
--color-sn-black-secondary: #121214;   /* Secondary background */
--color-sn-gold:            #c6a75e;   /* Primary brand gold */
--color-sn-gold-light:      #e0c97f;   /* Hover / highlight gold */
--color-sn-gray-dark:       #2a2a2e;
--color-sn-gray-medium:     #6b6b73;
--color-sn-gray-light:      #d1d1d6;
--color-sn-off-white:       #f5f5f7;
```

Usage in components: `bg-sn-black`, `bg-sn-black-secondary`, `text-sn-gold`, `bg-sn-gold`, `hover:bg-sn-gold-light`, etc.

### 9.3 UI Component Approach

- **Tailwind CSS** for all styling
- **shadcn/ui** as the base component library — unstyled-first, Tailwind-compatible, no Bootstrap or MUI
- Custom Sigma Nu theming applied on top via Tailwind v4 CSS token definitions in `globals.css`
- The site must not look like a template

## 10. Decision Log

A chronological record of every locked decision.

| # | Decision | Rationale |
|---|---|---|
| 1 | Next.js (App Router) as framework | SSR for tree performance, unified frontend/API, clean auth routing |
| 2 | Supabase as database and auth provider | Managed Postgres, built-in OAuth, storage — zero ops, right scale |
| 3 | React Flow for family tree | Purpose-built for node graphs, performant at 200–300 nodes; D3 rejected as overkill |
| 4 | Stripe for payments | Better DX, cheaper per transaction, cleaner checkout vs PayPal |
| 5 | Vercel for hosting | Native Next.js, auto SSL, GitHub-connected, zero ops |
| 6 | `big_id` self-referencing FK on `members` | Standard adjacency list — cleanest model for a rooted multi-child tree |
| 7 | Three-status user model: `pending` / `member` / `admin` | Pending can register but not access protected pages. Clean permission model. |
| 8 | Pledge class as text enum, no year field | Names never repeat (600 possible classes), no disambiguation needed |
| 9 | Full tree on load with minimap + search-to-fly | Maximizes impact for reunion context; usability solved by navigation tools |
| 10 | Simple event model in DB (not manual template editing) | One extra day of work saves code changes for every future event |
| 11 | `registration_guests` as separate table | Keeps exports clean and data queryable vs text blob |
| 12 | Google + Facebook + Apple OAuth supported | Requested by Rob, supported natively by Supabase Auth, not required for launch |
| 13 | 2–3 admin users | Supabase dashboard + app admin panel covers this comfortably |
| 14 | Profile fields mostly optional | Members control what they share; only name/email/pledge class required |
| 15 | Registration is final once submitted (no self-cancel) | Admin handles cancellations. Per Rob's answer on Q11. |
| 16 | Events managed dynamically — no hardcoded IDs or slugs anywhere in code | Any new event is a DB row; the 30th Anniversary Reunion is an ordinary record |
| 17 | `status` enum (`draft`/`published`/`archived`) replaces `registration_open` boolean | Enum supports three states cleanly; boolean only covered open/closed with no archived state |
| 18 | Canonical event routes: `/events/[id]` and `/events/[id]/register` | Per-event URLs enable multiple simultaneous events; `/register` is a legacy redirect |
| 19 | Admin event management at `/admin/events` | List, create, edit, and archive from one page; no hard deletes to preserve registration history |
| 20 | Homepage hero pulls next published event dynamically by `event_date ASC` | Multiple events can be published simultaneously; hero always shows the soonest upcoming one |
| 21 | Events can be archived but never deleted | Preserves all registration history and financial records tied to the event |
| 22 | Forgot password uses `supabase.auth.resetPasswordForEmail()` at `/auth/forgot-password` | No custom token handling — Supabase manages link generation and expiry |
| 23 | Reset password page reads recovery token from URL hash client-side via `onAuthStateChange` | Hash fragments are never sent to the server; must be handled in a client component |
| 24 | Duplicate email on signup intercepted and routed to password reset prompt | Avoids raw Supabase error strings leaking to users; guides them to the right action |
| 25 | Structured address fields (`street_address`, `city`, `state`, `zip`, `country`) added in Phase 13 — `home_address` preserved but deprecated | Structured fields enable clean display, better autocomplete UX, and future filtering by location |
| 26 | Google Places API loaded via dynamic script injection in `AddressAutocomplete.tsx` | Avoids Next.js `<Script>` loading order issues; degrades gracefully to plain text input if key absent |
| 27 | Privacy flags (`show_phone`, `show_address`, `show_birthday`) enforced at application query layer, not via PostgreSQL column-level security | PostgreSQL RLS cannot conditionally null a column based on another column's value; app-layer is simpler and equivalent. Admins bypass by using service-role queries. |
| 28 | Profile photos are cropped to 1:1 aspect ratio at 512×512px client-side (react-image-crop) before upload | Ensures consistent circular avatar display without server-side processing; 512px balances quality and storage cost |
| 29 | Birthday stored as PostgreSQL `date` type, serialised as `YYYY-MM-DD` string in JS | DATE avoids timezone ambiguity inherent in `timestamptz`; HTML date inputs return strings in this format natively |
| 30 | Referral system uses one-time UUID tokens with 7-day expiry stored in `referrals` table | UUIDs are unguessable, no collision risk, no sequential enumeration; 7-day window balances urgency vs convenience |
| 31 | Invite flow lives at `/join?token=[uuid]` — a public (unauthenticated) route | Invitees don't have accounts yet; token validation happens server-side via admin client |
| 32 | Invited person's email is set by the referrer and is read-only on the join form | Prevents email substitution attacks; referrer vouches for the specific person they're inviting |
| 33 | `referred_by` on members is admin-only visibility, enforced via `REVOKE SELECT (referred_by) ON members FROM authenticated` | Column-level revoke is the correct PostgreSQL mechanism for per-column access control; service role bypasses it |
| 34 | Token expiry maintenance via pg_cron nightly job — setup documented in migration file comments | pg_cron runs inside Postgres, no external scheduler needed; SQL and dashboard steps are in `20260405000005_referrals.sql` |

| 36 | Admin can cancel pending referrals at `/admin/referrals` — sets status to `expired`, no hard delete | Preserves audit trail; expired tokens show "already expired" on the invite link rather than "invalid" |
| 37 | Authenticated homepage lives at `/home`; post-login redirect updated from `/` to `/home` in proxy.ts | Landing page (`/`) remains public for event info; `/home` is the member dashboard with events, birthdays, announcements, and quick links |

| 39 | Birthdays this month on `/home` fetched via admin client (bypasses show_birthday RLS to get all consenting members), then month-filtered client-side, then sorted ascending by day within the month; filter requires `parts.length >= 3` to exclude malformed records before sorting | Admin client required since show_birthday=true members need to be cross-checked; client-side month filter avoids raw SQL substring on birthday text column; day sort uses `parseInt` so day 9 sorts before day 10 (numeric, not lexicographic); `parts.length < 3` guard ensures records without a day component never reach the sort comparator |

| 42 | Guest registration at `/events/[id]/register/guest` — public route, no auth, `member_id = null`, admin client for all DB writes | Allows non-alumni to register for events without creating an account; service role bypasses RLS that blocks anon inserts |

| 48 | ΣΝ logo in `(auth)/layout.tsx`, `auth/layout.tsx`, and `join/layout.tsx` wrapped in a Link to `/` | Ensures every page a logged-out user lands on has a navigable route back to the landing page |
| 49 | Post-registration guest management gated on `registration_open` only — no separate cutoff logic | Single boolean is sufficient; admin controls the window by toggling `registration_open` on the event |
| 50 | New guest payments write to `registration_payments` table, not `registrations.stripe_payment_id` | Stripe payment id column on registrations is a single value; multiple add-guest payments per registration require a separate payments table |
| 51 | `pending_guests jsonb` written before Stripe redirect, consumed and nulled by webhook on `checkout.session.completed` | Webhook is the only guaranteed-once handler; storing names server-side means no client round-trips needed after Stripe redirect. jsonb allows arbitrary array without schema changes. |

| 54 | Home page upcoming event cards extracted into `HomeEventsSection` client component | Cards need toggle state (expand/collapse manage panel) which requires `useState`; extracted to keep the server page itself as a pure async server component |
